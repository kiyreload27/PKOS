**Version:** 1.1

## 1. Executive Summary & Vision

PKOS is an intelligent, universally composable Knowledge Graph. It acts as an "operating system" for a user's knowledge, replacing traditional folder structures with an interconnected semantic graph that "remembers" context automatically.

**Why is the Architecture Frozen?**
The architecture is intentionally frozen because previous iterations accumulated unnecessary abstractions. Future development should prioritize completing features and refining UX over introducing new infrastructure. Any proposed architectural change must solve a demonstrated limitation that cannot be addressed within v1.1.

## 2. Architectural Invariants

Every new feature MUST satisfy all of the following:

- **The Event Store is the Source of Truth:** `PKOSEvent` holds the canonical history.
- **Read Models are Projections Only:** Projection tables exist solely for query performance. They may be deleted, rebuilt, and contain denormalized data. They are *never* considered canonical.
- **Domain is Pure:** `@pkos/domain` contains zero runtime dependencies (no Prisma, no Zod, no React).
- **AI Enriches Data, But Never Owns Canonical State:** AI suggestions require user approval or exist as provisional projections.
- **Every Write Emits Events:** Mutations are recorded via the Event Bus.
- **Every Projection Can Be Rebuilt From Events:** Dropping a projection table should only cost computation time, never data loss.
- **User-Facing Screens Read Projections:** UIs never aggregate live domain truth; they read fast projection tables.

## 3. Technology Stack & Repository Startup

- **Monorepo:** Turborepo + pnpm
- **Frontend:** Next.js 16 (App Router, Turbopack, Tailwind CSS, Lucide Icons)
- **Database:** PostgreSQL 16 (Managed via Prisma)
- **Background Tasks:** Redis 7 + BullMQ
- **Validation:** Zod (Isolated in `@pkos/validation`)

### Quick Start
```bash
pnpm install
docker compose up -d
pnpm db:push
pnpm dev
# In a separate terminal:
pnpm worker
```

## 4. Core Domain Objects

Everything in PKOS revolves around a lightweight `Entity`. Traits, resources, observations, and capabilities are attached, not baked in.

### Glossary
- **Capture:** The raw, immutable data entering the system (text, URL, file).
- **Resource:** A parsed, structured block of data from a Capture.
- **Entity:** The anchor. Represents a person, place, concept, or project.
- **Relationship:** A directed edge between two Entities.
- **Observation:** AI-generated insights or factual claims drawn from Entities.
- **Summary:** A concise abstract representing an Entity or Context.
- **Memory Card:** A projection representing curated knowledge/bookmarks generated by the system as evidence grows.

## 6. AI Constraints & Enrichment Pipeline

PKOS uses a strict priority to minimize hallucination: Deterministic Parsing -> Rules -> Graph Knowledge -> Semantic Search -> Embeddings -> LLM Reasoning. LLMs are only invoked when deterministic systems cannot produce reliable results.

### AI Constraints
**LLMs NEVER:**
- Invent Entities
- Delete Entities
- Merge Entities automatically
- Modify canonical data directly
- Bypass the Event Bus

**LLMs MAY:**
- Suggest merges (Placed in AI Inbox)
- Generate summaries
- Produce observations
- Recommend relationships
- Classify entities

## 7. Common Mistakes & Pitfalls

Future agents must avoid these common errors:

❌ **Query Prisma Domain models directly from React.**
✔ **Read Projection models.**

❌ **Add business logic to Next.js API routes.**
✔ **Put logic in the `@pkos/domain` package.**

❌ **Add synchronous AI processing in the web app.**
✔ **Emit a `PKOSEvent` and process it asynchronously in the worker.**

❌ **Create new hardcoded entity types in the database schema.**
✔ **Extend Entities using Domain Traits.**

❌ **Use `window.confirm()` for UI interactions.**
✔ **Build non-blocking, inline confirmation states to preserve headless QA compatibility.**

## 8. Monorepo Structure

```text
apps/
  ├── web/           (Next.js frontend. Projections and Actions)
  ├── worker/        (BullMQ background processor. Consumes the EventBus)
packages/
  ├── domain/        (Pure TS: Interfaces, Value Objects, Domain Services)
  ├── database/      (Prisma schema, client, Projection models)
  ├── validation/    (Zod schemas)
  ├── projections/   (Read-model materialization handlers)
  ├── workers/       (Job definitions and AI Enrichment pipelines)
  ├── plugins/       (Resource ingestion strategies)
  ├── api/           (Shared API utilities)
  ├── ui/            (Shared UI components)
```
